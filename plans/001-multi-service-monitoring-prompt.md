---

Currently, the Push to Talk service only accepts a single transcription server in its configuration. This feature would add support for specifying multiple servers in order of preference. Let's add functionality to accept multiple possible servers to communicate with, and use the health endpoint to check for their uptime. The list of servers should be ordered such that the user can specify them in the order of preference. The current inline health check at the start of recording is of limited value. We want to monitor proactively and select the right server before that event is reached. Access to and selection of a particular server should be gated by this new dedicated health monitor. The servers should be monitored for health at a particular interval, and that interval should also be configurable. The default interval should be 10 seconds with a timeout of 200 milliseconds. The monitor should run in parallel to the actual event handling. Should any given transcription service be unavailable for the time that transcription services are required, the next server in preference should be used. The idea of this feature is to allow for multiple possible services in the local network to be utilized when available, with slower fallbacks activated. On a longer timeline, this will help support the goal of falling back to local, albeit slower and lower quality, transcription services local to the device when all else fails.
